# 数据集定义

## 1. 背景

评测平台中的“数据集”贯穿 Trace 采样、人工标注、LLM 自动生成、质检审核、评测运行、回归验证和线上监控。为了避免把 Source Dataset、Golden Dataset、Benchmark、Validation Set、Test Set、Badcase Set 等行业概念都建成平台一等类型，langfusePro 第一版按实际使用场景将数据集类型收敛为三类：

- 评估集
- 评测集
- 回归集

其它行业常见概念不作为一等类型，而是通过来源、用途标签、环境标签、测试类型标签、质量等级、样本标签和版本状态表达。例如：Golden 表示质量等级，Benchmark 表示来源或标签，Badcase 表示失败样本标签和回归来源，train/dev/test 表示 split 或用途标签。

参考来源包括：

- [Google Machine Learning Crash Course](https://developers.google.com/machine-learning/crash-course/overfitting/dividing-datasets) 对 training / validation / test split 的定义与使用边界。
- [Hugging Face Datasets](https://huggingface.co/docs/datasets/index) 对数据集加载、split 和数据集仓库组织的通用实践。
- [Datasheets for Datasets](https://arxiv.org/abs/1803.09010) 对数据集动机、组成、采集、推荐用途、维护方式等文档化要求。
- LLM/Agent 评测领域关于 failure case、error analysis、production evaluation 和 regression 的实践。

## 2. 行业通用概念的收敛方式

| 行业概念 | 通用含义 | langfusePro 中的处理方式 |
| --- | --- | --- |
| Training Set | 用于训练或微调模型 | 作为用途标签或 split，不作为第一版数据集类型 |
| Validation / Dev Set | 开发过程中用于调参、调 Prompt、选模型 | 收敛为“评估集” |
| Test / Holdout Set | 最终验收用，尽量少用，避免被调参污染 | 收敛为“评测集”的冻结版本或保留 split |
| Golden Dataset | 高可信、已审核、可作为权威标准的数据 | 作为质量等级，常用于评测集或回归集 |
| Benchmark Dataset | 公开或内部标准化基准 | 作为来源或标签，可进入评测集 |
| Synthetic / LLM-generated Dataset | 由 LLM 自动生成或自动打标的数据 | 作为来源，进入三类数据集前需要抽检或质检 |
| Red-team / Safety Dataset | 安全、越权、注入、隐私等风险样本 | 作为测试类型标签或风险标签，通常进入评测集或回归集 |
| Badcase / Failure Case | 模型、Agent 或评测链路中的失败样本 | 作为失败样本标签和回归集的重要来源 |
| Production Monitoring Dataset | 线上抽样、用户反馈、告警、失败样本 | 作为来源，持续回流到评估集和回归集 |

## 3. 三类数据集定义

### 3.1 评估集

评估集用于开发和调优阶段，帮助团队快速判断某个模型、Prompt、Agent 编排、工具调用策略或评测指标是否值得继续推进。它更偏“迭代过程中的实验数据”，可以反复使用，但不能代表最终客观质量。

典型用途：

- Prompt 调优、模型初筛、参数和策略选择。
- 新评测指标、评分 Rubric、LLM Judge Prompt 的试验。
- 从线上 Trace、人工样本、LLM 自动生成样本中快速抽样验证。

数据来源：

- Langfuse Trace 抽样。
- 人工构造样本。
- LLM 自动生成样本。
- 线上反馈和少量 Badcase。
- 公开 Benchmark 的小规模试验子集。

治理要求：

- 可以高频使用和快速更新。
- 需要记录实验目标、样本来源、版本和适用范围。
- 不用于最终发布结论，避免“调到评估集上”导致过拟合。
- 可定期从线上反馈和 Badcase 中补充新样本，保持对真实问题的敏感度。

建议编码：

```text
EVALUATION_DEV
```

### 3.2 评测集

评测集用于正式评测运行和评测报告生成，面向模型、Prompt、Agent 配置或工具链版本之间的横向比较。它更偏“客观评价和结果发布”，需要比评估集更稳定、更可追溯。

典型用途：

- 离线评测运行。
- 多模型、多 Prompt、多 Agent 配置对比。
- 生成评测报告。
- 发布核心质量指标。
- 作为版本准入或阶段性验收依据。

数据来源：

- 已质检的人类标注样本。
- 高可信 Golden 样本。
- 公开或内部 Benchmark。
- 业务核心场景样本。
- 安全、红队、边界场景样本。

治理要求：

- 必须绑定评测目标、指标口径、评分标准和适用范围。
- 需要版本化，重要版本应冻结。
- 与评估集隔离，避免在正式评测集上反复调参。
- 报告中应按场景、难度、风险级别、来源、Badcase 标签拆分指标，避免只看平均分。

建议编码：

```text
EVALUATION
```

### 3.3 回归集

回归集用于持续验证“核心能力不退化、历史问题不复发”。它应长期稳定维护，覆盖核心路径、历史故障、线上高频问题、高风险 Badcase 和发布门禁样本。

典型用途：

- 每次模型、Prompt、Agent 编排、工具调用策略变更后的回归测试。
- 发布前质量门禁。
- 验证历史 Badcase 是否已修复且不复发。
- CI/CD 或定时评测任务。

数据来源：

- 历史 Badcase。
- 线上告警、用户投诉、人工反馈。
- 生产事故复盘样本。
- 高风险安全样本和权限边界样本。
- 核心业务流程的 Basecase。

治理要求：

- 样本数量可以不大，但必须稳定、代表核心风险。
- 每条样本都应记录入选原因、期望行为、历史背景、严重级别和修复状态。
- 重要回归样本应冻结，变更需要审批。
- Badcase 修复后不应删除，而应沉淀为回归样本，防止问题复发。

建议编码：

```text
REGRESSION
```

## 4. Badcase 定义与治理

### 4.1 Badcase 是什么

Badcase 是模型、Agent、工具调用链路或评测链路未达到预期的失败样本。行业里也常称为 failure case、error case、hard example 或 misprediction case。Badcase 的价值不只是“记录错误”，更重要的是支持错误分析、根因定位、修复验证和回归防护。

在 langfusePro 中，Badcase 不建议作为独立的一等数据集类型，而建议作为样本标签和回归集的重要来源：

```text
Badcase = 样本标签 / 失败案例资产
回归集 = Badcase 沉淀后的长期验证集合
```

### 4.2 Badcase 常见来源

| 来源 | 示例 |
| --- | --- |
| 线上 Trace | 低分回答、工具调用失败、用户追问、人工差评 |
| 评测运行 | 指标不达标、Judge 判失败、格式校验失败 |
| 人工质检 | 标注审核不通过、专家判断不符合预期 |
| 用户反馈 | 投诉、纠错、显式负反馈 |
| 安全测试 | Prompt injection、越权调用、敏感信息泄露 |
| 系统监控 | 超时、异常、重试失败、外部依赖错误 |

### 4.3 Badcase 分类建议

| 分类 | 说明 |
| --- | --- |
| 事实性错误 | 回答与事实、上下文或知识库不一致 |
| 幻觉 | 模型编造不存在的信息、引用、工具结果或业务规则 |
| 指令遵循失败 | 没有遵守用户约束、系统规则、格式要求 |
| 工具调用错误 | 工具选错、参数错误、调用顺序错误、漏调用 |
| 检索失败 | 没有召回关键上下文，或使用了错误上下文 |
| 多轮状态错误 | 遗忘上下文、状态串线、错误继承历史信息 |
| 权限与安全错误 | 越权访问、敏感信息泄露、绕过安全策略 |
| 输出格式错误 | JSON/schema 不合法、字段缺失、类型错误 |
| 鲁棒性问题 | 边界输入、噪声输入、长上下文、异常语言导致失败 |
| 性能与稳定性问题 | 超时、重试失败、外部依赖失败、成本异常 |
| 评测链路问题 | 标准答案错误、Judge 误判、数据泄漏、样本污染 |

### 4.4 Badcase 生命周期

```text
发现 Badcase
  -> 归一化样本和上下文
  -> 标注期望行为与失败现象
  -> 归因分类和严重级别评定
  -> 进入修复队列
  -> 修复后复测
  -> 沉淀到回归集
  -> 定期回归和复发监控
```

### 4.5 Badcase 样本字段

| 字段 | 说明 |
| --- | --- |
| badcaseId | 失败样本唯一标识 |
| projectId | 所属项目 |
| traceId | 对应 Langfuse Trace |
| source | 来源：线上、评测、质检、用户反馈、安全测试 |
| input | 用户输入或任务输入 |
| context | 检索上下文、工具上下文、多轮历史 |
| actualOutput | 实际输出 |
| expectedBehavior | 期望行为或标准答案 |
| failureType | 失败类型 |
| severity | 严重级别 |
| rootCause | 根因分析 |
| owner | 责任人 |
| fixStatus | 待分析、修复中、已修复、暂不处理 |
| regressionCandidate | 是否建议进入回归集 |
| regressionDatasetId | 关联的回归集 |

## 5. 三类数据集对比

| 类型 | 核心目的 | 使用频率 | 稳定性 | 是否用于正式报告 | 是否承接 Badcase |
| --- | --- | --- | --- | --- | --- |
| 评估集 | 快速实验和调优 | 高 | 低到中 | 否 | 可少量承接，用于快速验证 |
| 评测集 | 正式评价和横向比较 | 中 | 中到高 | 是 | 可按场景纳入已确认 Badcase |
| 回归集 | 防止核心能力退化和历史问题复发 | 高，通常自动化 | 高 | 可作为发布门禁报告 | 是，Badcase 的主要沉淀位置 |

## 6. 标签体系

数据集主类型只解决“这个数据集主要用于什么场景”的问题；标签用于解决“这个数据集适用于什么环境、什么测试强度、什么风险场景、什么业务域”的问题。一个数据集只能有一个主类型，但可以拥有多个标签。

### 6.1 标签维度

| 标签维度 | 建议编码 | 说明 | 示例 |
| --- | --- | --- | --- |
| 环境标签 | `env` | 标识数据集适用或来源环境 | 开发、测试、生产 |
| 测试类型标签 | `testType` | 标识评测强度或测试目标 | 冒烟、标准、红队/安全 |
| 业务场景标签 | `businessScenario` | 标识业务域或任务类型 | 客服问答、工单处理、知识库检索、Agent 工具调用 |
| 风险标签 | `riskType` | 标识风险或安全类别 | 越权、注入攻击、隐私泄露、幻觉 |
| Badcase 标签 | `badcaseType` | 标识失败样本类型 | 工具调用错误、检索失败、格式错误 |
| 难度标签 | `difficulty` | 标识样本复杂度 | 简单、中等、困难、长尾 |

### 6.2 环境标签

| 标签 | 编码建议 | 说明 |
| --- | --- | --- |
| 开发 | `DEV` | 用于开发调试、Prompt 试验、快速验证，通常对应评估集 |
| 测试 | `TEST` | 用于测试环境、预发布环境或准入验证，常用于评测集和回归集 |
| 生产 | `PROD` | 来自生产环境或用于生产质量监控，通常需要脱敏、审计和更严格权限 |

说明：

- “生产”标签表示样本来自真实生产流量或用于生产质量监控，不代表该数据集可以直接在线上执行。
- 生产来源样本进入评测或回归前，应完成脱敏、授权和保留周期确认。

### 6.3 测试类型标签

| 标签 | 编码建议 | 说明 |
| --- | --- | --- |
| 冒烟 | `SMOKE` | 少量核心样本，用于快速验证链路是否可用 |
| 标准 | `STANDARD` | 覆盖常规业务场景，用于常规评测、版本比较和报告发布 |
| 红队/安全 | `REDTEAM_SECURITY` | 覆盖攻击、越权、隐私、敏感输出和工具滥用等风险场景 |

使用建议：

- 冒烟标签适合放在评估集或回归集中，用于快速检查评测链路、模型调用链路和核心业务链路。
- 标准标签适合作为评测报告的主体样本。
- 红队/安全标签通常需要更严格权限控制，并在报告中单独展示，不建议与普通场景简单求平均分。

### 6.4 标签与三类数据集的关系

| 主类型 | 常见环境标签 | 常见测试类型标签 | 说明 |
| --- | --- | --- | --- |
| 评估集 | 开发、测试 | 冒烟、标准 | 支持快速实验，可频繁调整标签组合 |
| 评测集 | 测试、生产 | 标准、红队/安全 | 支持正式评测和报告，标签用于指标分层展示 |
| 回归集 | 测试、生产 | 冒烟、标准、红队/安全 | 支持发布门禁和问题复发监控，Badcase 标签尤其重要 |

## 7. 推荐元数据

每个数据集至少应包含以下元数据：

| 字段 | 说明 |
| --- | --- |
| datasetId | 数据集唯一标识 |
| name | 数据集名称 |
| type | `EVALUATION_DEV`、`EVALUATION`、`REGRESSION` 三选一 |
| projectId | 所属项目，权限隔离边界 |
| source | Trace、导入、Benchmark、LLM 生成、人工构造、Badcase 回流等 |
| version | 数据集版本 |
| status | 草稿、标注中、质检中、已发布、已冻结、已归档 |
| split | train、dev、test、holdout，可选 |
| environmentTags | 环境标签，例如开发、测试、生产 |
| testTypeTags | 测试类型标签，例如冒烟、标准、红队/安全 |
| qualityLevel | 未审核、抽检通过、全量质检、Golden、冻结 |
| tags | 业务场景、风险类型、Badcase 类型、难度等扩展标签 |
| sampleCount | 样本数量 |
| labelSchema | 标签 schema、评分 Rubric、答案格式 |
| owner | 数据负责人 |
| reviewPolicy | 抽检、双人审核、专家审核等质检策略 |
| refreshPolicy | 手动、按版本、定期、线上回流 |
| compliance | 授权、脱敏、保留周期和访问限制 |

## 8. 推荐流转关系

```text
线上 Trace / 业务导入 / Benchmark / LLM 生成 / 用户反馈
  -> 样本池
  -> 标注与质检
  -> 环境标签 / 测试类型标签 / 业务标签 / 风险标签
  -> 评估集：开发调优、快速实验
  -> 评测集：正式评测、报告发布
  -> Badcase 分析：失败样本归因、严重级别、修复状态
  -> 回归集：历史问题沉淀、发布门禁、持续监控
  -> 新一轮线上监控和样本回流
```

流转原则：

- 评估集服务迭代，允许频繁调整。
- 评测集服务正式评价，应版本化、可追溯、必要时冻结。
- 回归集服务长期质量防线，重点吸收 Badcase、Basecase 和高风险场景。
- Badcase 必须先完成归因和期望行为定义，再进入回归集。
- 标签应支持多选和组合筛选，例如“测试环境 + 冒烟”、“生产来源 + 红队/安全”、“回归集 + 工具调用错误 Badcase”。

## 9. 与平台对象的关系

当前平台对象链路建议统一为：

```text
项目
  -> Trace / 外部导入 / LLM 生成 / 用户反馈
  -> 样本池
  -> 标注与质检
  -> 评估集 / 评测集 / 回归集
  -> 评测运行
  -> 评测报告
  -> Badcase 回流
```

三类数据集在平台中的定位：

| 平台类型 | 建议编码 | 定位 |
| --- | --- | --- |
| 评估集 | `EVALUATION_DEV` | 开发调优和实验验证 |
| 评测集 | `EVALUATION` | 正式评测和报告发布 |
| 回归集 | `REGRESSION` | Badcase、Basecase 和发布门禁 |

## 10. 设计建议

- 平台一等数据集类型只保留三类：评估集、评测集、回归集。
- 平台应支持给不同数据集打多维标签，至少覆盖环境标签和测试类型标签。
- 环境标签建议内置开发、测试、生产；测试类型标签建议内置冒烟、标准、红队/安全。
- Badcase 不做一等数据集类型，而是样本标签、分析对象和回归集来源。
- Golden、Benchmark、Safety、LLM-generated、Production Monitoring 作为来源、标签、质量等级或治理状态表达。
- train/dev/test 作为 split 字段，不与业务类型混在同一层。
- 评测报告必须按场景、难度、风险标签、Badcase 类型拆分指标，避免平均分掩盖问题。
- 回归集应支持自动化运行，并能追踪 Badcase 是否复发。
